# Requirements Document

## Introduction

Рефакторинг навигации фронтенд-приложения MEXC Spread Monitor. Текущая архитектура использует модальные окна (overlay-диалоги), управляемые через `useState`-флаги в `App.tsx`, для отображения разделов: Trading Admin, Spread Capture, AsterDEX, Арбитраж, Futures Arb, История спреда, Алерты. Требуется перевести каждый раздел на отдельную страницу (маршрут) с клиентской маршрутизацией, боковой навигацией и поддержкой URL-адресов.

## Glossary

- **Router**: Библиотека клиентской маршрутизации (react-router-dom), обеспечивающая переключение между страницами без перезагрузки браузера
- **Layout**: Корневой компонент-обёртка, содержащий боковую навигацию и область отображения текущей страницы
- **Sidebar**: Боковая панель навигации с перечнем всех доступных разделов приложения
- **Route**: Уникальный URL-путь, соответствующий определённому разделу приложения
- **Spread_Monitor**: Главная страница приложения, отображающая таблицу/плитки спредов с фильтрами и переключателем бирж
- **Navigation_State**: Глобальное состояние приложения (выбранная биржа, рынок, фильтры), сохраняемое при переходах между страницами
- **Page_Component**: React-компонент, представляющий отдельную страницу-раздел приложения

## Requirements

### Requirement 1: Установка и настройка клиентской маршрутизации

**User Story:** Как разработчик, я хочу иметь клиентскую маршрутизацию в приложении, чтобы каждый раздел был доступен по уникальному URL-адресу.

#### Acceptance Criteria

1. THE Router SHALL обеспечивать навигацию между страницами без полной перезагрузки браузера, используя библиотеку react-router-dom версии 6.x
2. WHEN пользователь вводит URL маршрута в адресную строку и нажимает Enter, THE Router SHALL отобразить соответствующую страницу в течение 1 секунды без перезагрузки всего приложения
3. WHEN пользователь нажимает кнопку «Назад» в браузере, THE Router SHALL вернуть предыдущую страницу из истории навигации с сохранением состояния компонента
4. THE Router SHALL использовать режим HTML5 History API (BrowserRouter), формируя URL без hash-фрагментов (например `/trading`, а не `/#/trading`)

### Requirement 2: Определение маршрутов для каждого раздела

**User Story:** Как пользователь, я хочу, чтобы каждый раздел приложения имел свой URL-адрес, чтобы я мог добавлять страницы в закладки и делиться ссылками.

#### Acceptance Criteria

1. THE Router SHALL определить маршрут `/` для страницы Spread_Monitor (главная)
2. THE Router SHALL определить маршрут `/trading` для страницы Trading Admin
3. THE Router SHALL определить маршрут `/spread-capture` для страницы Spread Capture
4. THE Router SHALL определить маршрут `/asterdex` для страницы AsterDEX
5. THE Router SHALL определить маршрут `/arbitrage` для страницы Арбитраж
6. THE Router SHALL определить маршрут `/futures-arb` для страницы Futures Arb
7. THE Router SHALL определить маршрут `/spread-history` для страницы История спреда
8. THE Router SHALL определить маршрут `/alerts` для страницы Алерты
9. WHEN пользователь переходит по несуществующему маршруту, THE Router SHALL перенаправить на главную страницу `/`
10. WHEN пользователь переходит на маршрут напрямую через адресную строку (deep link), THE Router SHALL отобразить соответствующую страницу без промежуточных редиректов
11. THE Router SHALL синхронизировать URL в адресной строке браузера при каждом программном переходе между страницами

### Requirement 3: Боковая навигационная панель

**User Story:** Как пользователь, я хочу иметь боковое меню со списком всех разделов, чтобы быстро переключаться между страницами.

#### Acceptance Criteria

1. THE Layout SHALL отображать Sidebar с перечнем следующих разделов приложения: Spread Monitor (спот/фьючерсы/базис), Trading Admin, Spread Capture, AsterDEX, Арбитраж, Futures Arb, История спреда, Алерты
2. THE Sidebar SHALL выделять текущий активный раздел отличающимся фоновым цветом и жирным начертанием текста, так что активный пункт визуально отличим от неактивных без взаимодействия
3. WHEN пользователь нажимает на пункт меню в Sidebar, THE Layout SHALL открыть соответствующий раздел без полной перезагрузки страницы (client-side переход)
4. THE Sidebar SHALL отображать иконку и текстовое название длиной не более 30 символов для каждого пункта меню
5. WHEN пользователь нажимает кнопку сворачивания в Sidebar, THE Layout SHALL переключить Sidebar в режим «только иконки» с шириной не более 64 px
6. WHILE Sidebar находится в свёрнутом состоянии, THE Layout SHALL отображать только иконки пунктов меню и показывать всплывающую подсказку с названием раздела при наведении курсора на иконку в течение не менее 300 мс

### Requirement 4: Адаптивный макет с боковой навигацией

**User Story:** Как пользователь, я хочу, чтобы интерфейс корректно отображался на разных размерах экрана, чтобы пользоваться приложением на любом устройстве.

#### Acceptance Criteria

1. THE Layout SHALL размещать Sidebar фиксированной ширины слева, а область контента страницы — справа, занимая всё оставшееся горизонтальное пространство
2. WHILE ширина viewport менее 768px, THE Layout SHALL скрывать Sidebar и отображать кнопку-гамбургер в верхней части экрана для его вызова
3. WHILE ширина viewport 768px и более, THE Layout SHALL отображать Sidebar постоянно без кнопки-гамбургер
4. WHEN пользователь нажимает кнопку-гамбургер на экране шириной менее 768px, THE Sidebar SHALL появляться поверх контента как выдвижная панель с полупрозрачным затемняющим фоном (backdrop) позади неё
5. WHEN пользователь нажимает на backdrop или повторно на кнопку-гамбургер, THE Sidebar SHALL закрываться и возвращать интерфейс к состоянию со скрытой панелью
6. THE Layout SHALL занимать полную высоту viewport (100vh) без вертикальной прокрутки на уровне Layout, при этом вертикальная прокрутка допускается только внутри области контента

### Requirement 5: Сохранение глобального состояния при навигации

**User Story:** Как пользователь, я хочу, чтобы выбранная биржа, рынок и фильтры сохранялись при переходе между страницами, чтобы не настраивать их заново.

#### Acceptance Criteria

1. WHILE пользователь переходит между страницами приложения, THE Navigation_State SHALL сохранять текущее значение выбранной биржи без изменений
2. WHILE пользователь переходит между страницами приложения, THE Navigation_State SHALL сохранять текущее значение выбранного рынка (spot, futures или cross) без изменений
3. WHILE пользователь переходит между страницами приложения, THE Navigation_State SHALL сохранять текущие значения фильтров (котировка, минимальный спред, минимальный объём, поисковая строка) без изменений, так что при возврате к основной таблице Spread_Monitor отображаются те же отфильтрованные данные
4. WHEN пользователь перезагружает страницу в браузере, THE Navigation_State SHALL восстановить выбранную биржу и рынок, применяя следующий приоритет: URL-параметры (query string) имеют приоритет над localStorage; если URL-параметры отсутствуют, используются значения из localStorage
5. IF при загрузке страницы отсутствуют сохранённые значения и в URL, и в localStorage, THEN THE Navigation_State SHALL использовать значения по умолчанию: биржа — "mexc", рынок — "spot", котировка — "Все", минимальный спред — 0, минимальный объём — 0, поисковая строка — пустая
6. WHEN пользователь изменяет биржу или рынок, THE Navigation_State SHALL синхронно записать новое значение в localStorage в течение 1 секунды, чтобы оно было доступно при последующей перезагрузке
7. THE Navigation_State SHALL предоставлять текущие значения биржи, рынка и фильтров всем компонентам приложения через React Context, так что любой компонент, отображающий данные снимка, использует актуальную биржу и рынок для запросов к API

### Requirement 6: Преобразование модальных компонентов в страницы

**User Story:** Как разработчик, я хочу, чтобы каждый модальный компонент стал полноценной страницей, чтобы упростить архитектуру и улучшить навигацию.

#### Acceptance Criteria

1. THE Page_Component для Trading Admin SHALL отображать полный функционал TradingAdminModal (статус движка, запуск/остановка, kill switch, runtime-настройки) как содержимое страницы без модального обрамления (overlay, backdrop, кнопка закрытия)
2. THE Page_Component для Spread Capture SHALL отображать полный функционал SpreadCapturePanel как содержимое страницы без модального обрамления
3. THE Page_Component для AsterDEX SHALL отображать полный функционал AsterDexPanel как содержимое страницы без модального обрамления
4. THE Page_Component для Арбитраж SHALL отображать полный функционал ArbitragePanel как содержимое страницы без модального обрамления
5. THE Page_Component для Futures Arb SHALL отображать полный функционал FuturesArbPanel как содержимое страницы без модального обрамления
6. THE Page_Component для История спреда SHALL отображать полный функционал CrossSpreadHistoryChart как содержимое страницы без модального обрамления
7. THE Page_Component для Алерты SHALL отображать полный функционал AlertsSettingsPanel как содержимое страницы без модального обрамления
8. WHEN модальный компонент преобразован в Page_Component, THE Page_Component SHALL сохранить весь существующий функционал (кнопки, формы, таблицы, графики) без потерь
9. THE Page_Component SHALL адаптировать свой layout к полной ширине области контента (вместо фиксированной ширины модального окна)
10. WHEN Page_Component монтируется, THE Page_Component SHALL самостоятельно загружать необходимые данные с API (без зависимости от родительского компонента App)

### Requirement 7: Главная страница Spread Monitor

**User Story:** Как пользователь, я хочу, чтобы главная страница оставалась монитором спредов с таблицей/плитками, чтобы основной функционал был доступен сразу при открытии приложения.

#### Acceptance Criteria

1. WHEN пользователь открывает приложение по корневому URL `/`, THE Spread_Monitor SHALL отображаться как страница по умолчанию без дополнительной навигации или редиректа
2. THE Spread_Monitor SHALL предоставлять следующие элементы управления: переключатель режима отображения (таблица/плитки), фильтры (котировка, минимальный спред, минимальный объём, поиск по символу), сортировку по колонкам, переключатель рынков (spot/futures/cross), переключатель бирж (ExchangeSwitcher) и управление автообновлением с настраиваемым интервалом
3. THE Spread_Monitor SHALL отображать переключатель бирж (ExchangeSwitcher) в области контента страницы, сгруппированный по категориям (CEX, DEX), с визуальным выделением активной биржи
4. THE Spread_Monitor SHALL поддерживать модальные окна (ChartModal, SpreadChartModal, DomModal) как overlay поверх страницы, не блокируя доступ к основному контенту после закрытия модального окна
5. IF загрузка данных с API при открытии страницы завершается ошибкой, THEN THE Spread_Monitor SHALL отобразить сообщение об ошибке и предоставить возможность повторной загрузки без перезагрузки страницы
6. WHEN пользователь открывает страницу впервые (без сохранённых настроек в localStorage), THE Spread_Monitor SHALL отображать данные в режиме таблицы с сортировкой по спреду (bps) по убыванию

### Requirement 8: Конфигурация Vite для клиентской маршрутизации

**User Story:** Как разработчик, я хочу, чтобы dev-сервер и production-сборка корректно обрабатывали клиентские маршруты, чтобы прямой переход по URL работал без ошибки 404.

#### Acceptance Criteria

1. WHEN Vite dev-сервер получает GET-запрос на путь, который не соответствует существующему статическому файлу и не начинается с `/api/`, THE Vite SHALL возвращать содержимое `index.html` с HTTP-статусом 200 (SPA fallback)
2. WHEN production-сборка обслуживается через FastAPI и поступает GET-запрос на путь, который не начинается с `/api/` и не соответствует существующему статическому файлу (JS, CSS, изображения, шрифты), THEN THE FastAPI SHALL возвращать содержимое `index.html` с HTTP-статусом 200
3. IF запрос начинается с `/api/` и не соответствует ни одному зарегистрированному API-эндпоинту, THEN THE FastAPI SHALL возвращать ответ с HTTP-статусом 404 без SPA fallback
4. WHEN Vite dev-сервер получает запрос, начинающийся с `/api/`, THE Vite SHALL проксировать запрос на backend-сервер FastAPI (по умолчанию `http://127.0.0.1:8000`) без применения SPA fallback
